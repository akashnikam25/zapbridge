## Problem Statement

interview at Zapier. Each design decision must have a WHY you can articulate under

**Interview target:** Zapier Senior Backend Engineer — Enterprise Integrations Team

**Three questions you must answer with real code:**
1. Most complex REST API integration you built? → GitHub Issues API (pagination + backoff)
2. OAuth 2.0 in production? → GitHub OAuth (encrypt → store → refresh → revoke)
3. AI workflow you built? → Thin harness: single `.create()` call, validate output, log decision

---

## What Makes This Production-Level (Not Just "Working")

The difference between a personal project and production code is not features — it's
the reasoning embedded in each decision. Here's what separates them for each slice:

| Pattern | Personal project | Production |
|---|---|---|
| Webhook validation | Skip or check header exists | HMAC-SHA256 + `compare_digest` (timing-safe) |
| Idempotency | Hope GitHub doesn't retry | Redis `SETNX` — atomic, TTL-bound, logs duplicates |
| Token storage | Plain text in DB | Fernet AES-128-CBC, key from env var |
| AI calls | `response.text` used directly | Validate length + content before acting |
| Error handling | `print(e)` or pass | Structured log with context, job retry, dead letter |
| HTTP responses | Return 200 when done | Return 200 immediately, process async |

### Slice 1: Webhook → Slack (No security yet) — ~2 hours

**What you build:**

# POST /webhook — receives any GitHub event, posts to Slack
@app.post("/webhook")
async def receive_webhook(request: Request):

before adding complexity."

**The `post_to_slack()` implementation:**
```python
# app/slack.py
import httpx
from app.config import settings

# One sync function — works for both FastAPI endpoint (Slice 1) and RQ worker (Slice 4+).
# Blocks the event loop for ~100ms in Slice 1, which is invisible at demo scale.
def post_to_slack(message: str) -> None:
    with httpx.Client(timeout=10.0) as client:
        resp = client.post(settings.SLACK_WEBHOOK_URL, json={"text": message})
    if resp.status_code != 200:
        raise RuntimeError(f"Slack rejected message: {resp.status_code} {resp.text}")
```

### Slice 2: HMAC-SHA256 Signature Validation — ~1 hour

**What you build:**
```python
# app/webhooks/validator.py
import hmac, hashlib

def validate_signature(payload: bytes, signature_header: str, secret: str) -> bool:
    if not signature_header.startswith("sha256="):
        return False
    expected = "sha256=" + hmac.new(
        secret.encode(), payload, hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(expected, signature_header)  # Timing-safe
```

**Interview story for `compare_digest`:** "A naive `==` comparison short-circuits on the
first mismatching byte. An attacker can measure how long your server takes to reject
different payloads and reconstruct the valid signature bit by bit. `compare_digest`
always takes the same time regardless of where the mismatch is — it's timing-safe."

**Done when:** Remove your webhook secret → POST is rejected with 401. Add it back → accepted.

### Slice 3: Redis Idempotency (SETNX) — ~1 hour

**What you build:**
```python
# app/webhooks/receiver.py
IDEMPOTENCY_TTL = 86400  # 24 hours

def is_duplicate(delivery_id: str) -> bool:
    key = f"webhook:{delivery_id}"
    # SET NX EX is fully atomic: set-if-not-exists + expiry in one round-trip.
    # SETNX + EXPIRE is NOT atomic — if the process dies between them, the key
    # has no TTL and that delivery ID is locked out forever (memory leak + permanent block).
    already_seen = not redis_conn.set(key, 1, nx=True, ex=IDEMPOTENCY_TTL)
    return already_seen
```

**Interview story for SET NX EX:** "GitHub retries webhooks if your server doesn't respond
within 10 seconds — or if it gets a non-2xx. Without idempotency, a temporary outage
means every event gets processed twice. I use `SET key 1 NX EX 86400` — one atomic
Redis command. NX means 'only set if not exists', EX sets the TTL in the same operation.
The older pattern (`SETNX` + `EXPIRE`) isn't crash-safe: if your process dies between
those two commands, the key has no TTL and that delivery ID is blocked forever. The
atomic form eliminates that entire failure mode."

**Done when:** Manually resend the same GitHub webhook delivery → second POST is
rejected with `{"status": "already_processed"}`.

### Slice 4: RQ Worker + Async Processing — ~2 hours

**What you build:**
```python
# app/webhooks/receiver.py — return 200 immediately
from rq import Retry
job = queue.enqueue(
    process_github_event, event_type, payload,
    job_timeout=120,
    retry=Retry(max=3, interval=[10, 30, 60]),  # 3 retries with backoff; after 3 failures → DLQ
    failure_ttl=86400,   # Keep failed jobs for 24h for inspection, then auto-clean
)
return {"status": "queued", "job_id": job.id}

# app/workers/processor.py — runs in separate process (synchronous context)
def process_github_event(event_type: str, payload: dict) -> None:
    message = format_message(event_type, payload)  # Claude goes here in Slice 5
    post_to_slack(message)  # RQ workers are synchronous — post_to_slack is sync
```

**`format_message()` for pre-Claude Slack messages (Slices 1-4):**
```python
def format_message(event_type: str, payload: dict) -> str:
    repo = payload.get("repository", {}).get("full_name", "")
    action = payload.get("action", "")
    return f"*[{repo}]* GitHub `{event_type}` event — action: `{action}`"
```
This gets replaced by Claude's summary in Slice 5.

**Run worker:** `rq worker zapbridge` (separate terminal)

**Interview story:** "The webhook receiver must return 200 within 10 seconds or GitHub
considers it failed and retries. If Claude takes 3 seconds, or Slack is slow, you'd
fail. So the receiver does ONE thing: validate the signature, check idempotency, enqueue.
The worker does the heavy work. Classic producer-consumer."

**Done when:** Kill the worker → webhooks still queue. Start worker → messages drain.

### Slice 5: Claude AI Summarization — ~1 hour

**What you build:**
```python
# app/workers/processor.py
def _summarize_event(event_type: str, payload: dict) -> str:
    response = anthropic_client.messages.create(
        model="claude-haiku-4-5-20251001",  # Fast + cheap for summaries
        max_tokens=150,
        messages=[{"role": "user", "content": build_prompt(event_type, payload)}],
        timeout=30.0,  # Fail fast — don't let Anthropic slowness block the worker for 120s
    )
    summary = response.content[0].text.strip()

    # Validate output — never post garbage to Slack
    if len(summary) < 10:
        raise ValueError(f"Claude returned unusable summary: {summary!r}")

    logger.info("claude_summary generated", extra={"len": len(summary)})
    return summary
```

**The `build_prompt()` function:**
```python
def build_prompt(event_type: str, payload: dict) -> str:
    repo = payload.get("repository", {}).get("full_name", "unknown/repo")
    sender = payload.get("sender", {}).get("login", "unknown")
    # Truncate payload to avoid token waste — first 2000 chars covers all useful fields
    return f"""Summarize this GitHub {event_type} event for a Slack notification.
Be specific: what changed, who changed it, why it matters.
Max 2 sentences. No filler.

Repository: {repo}
Sender: {sender}
Event: {json.dumps(payload)[:2000]}"""
```

**Interview story for validation:** "AI output is non-deterministic. The guard rails are:
max_tokens to bound cost, a minimum length check to catch empty responses, and logging
every summary so you can audit what got posted. The AI call is a single `.create()` —
my code controls the flow, not a framework."

**Interview story for model choice:** "Haiku for summaries — it's 10x cheaper than Sonnet
and the quality is identical for 2-sentence event summaries. Sonnet is overkill here."

**Done when:** GitHub PR event arrives → Slack shows Claude-written summary, not raw JSON.

### Slice 6: GitHub OAuth + Fernet Token Encryption — ~3 hours

**What you build:**
```python
# app/auth/tokens.py
from cryptography.fernet import Fernet
_fernet = Fernet(settings.FERNET_KEY.encode())

def encrypt(token: str) -> str:
    return _fernet.encrypt(token.encode()).decode()

def decrypt(encrypted: str) -> str:
    return _fernet.decrypt(encrypted.encode()).decode()

# app/auth/oauth.py
# /auth/login → generate state UUID → store in Redis (TTL 600s) → redirect to GitHub
# /auth/callback → GETDEL state from Redis → verify → exchange code → encrypt tokens → store in DB
# /auth/disconnect → revoke GitHub token → delete user row

# CSRF state via Redis (not in-memory — survives restarts and works with multiple workers)
CSRF_STATE_TTL = 600  # 10 minutes

def store_oauth_state(state: str) -> None:
    redis_conn.set(f"oauth_state:{state}", 1, ex=CSRF_STATE_TTL)

def consume_oauth_state(state: str) -> bool:
    # GETDEL: atomic get-and-delete — prevents replay and race conditions
    return redis_conn.getdel(f"oauth_state:{state}") is not None
```

**Interview story for CSRF state:** "I store the OAuth state parameter in Redis with a
10-minute TTL, not in-memory. In-memory state fails with multiple workers (each worker
has its own dict) and on any process restart. Redis is already in the stack — one
`SET oauth_state:{uuid} 1 EX 600` on login, one atomic `GETDEL` on callback. The GETDEL
is important: it verifies AND deletes in one operation, preventing replay attacks."

**Database model:**
```python
class User(Base):
    __tablename__ = "users"
    id: Mapped[int] = mapped_column(primary_key=True)
    github_id: Mapped[int] = mapped_column(unique=True)
    github_login: Mapped[str] = mapped_column(String(255))
    access_token_enc: Mapped[str] = mapped_column(Text)      # Encrypted
    refresh_token_enc: Mapped[str | None] = mapped_column(Text, nullable=True)
    token_expires_at: Mapped[datetime | None] = mapped_column(DateTime, nullable=True)
```

**Interview story for Fernet:** "I encrypt tokens at rest using Fernet (AES-128-CBC with
HMAC-SHA256). The key lives only in an env var — not in the DB, not in code. If someone
dumps the database, they get ciphertext. If they also get the key (env var), you have
bigger problems — rotate the key."

**Interview story for refresh token logic:** "GitHub OAuth tokens don't expire — I know this.
I implemented the refresh pattern anyway as a demonstration that applies to every real-world
OAuth provider: Google (1h expiry), Salesforce (2h), Slack (varies). The function is named
`get_or_refresh_token()` with a clear comment: '# NOTE: GitHub tokens never expire; this
branch only runs with providers that issue short-lived tokens (Google, Salesforce, Slack).'
If `expires_at` is null (GitHub), return the token immediately. If the token is within
5 minutes of expiry, call the refresh endpoint, update the DB, return the new token.
The caller never knows a refresh happened. I preempt the question by telling this story
before the interviewer can catch me on it."

**Done when:** Visit `/auth/login` → GitHub OAuth → tokens stored encrypted in DB → `/auth/disconnect` → row deleted, GitHub token revoked.

### Slice 7: GitHub Issues API (Pagination + Backoff) — ~2 hours

**What you build:**
```python
# app/github.py
def fetch_all_issues(token: str, repo: str) -> list[dict]:
    headers = {"Authorization": f"Bearer {token}", "Accept": "application/vnd.github+json"}
    all_issues, page = [], 1

    while True:
        data = fetch_with_retry(
            f"https://api.github.com/repos/{repo}/issues?state=open&per_page=30&page={page}",
            headers
        )
        all_issues.extend(data)
        if len(data) < 30:
            break  # Last page
        page += 1

    return all_issues

def fetch_with_retry(url: str, headers: dict, max_retries: int = 5) -> list:
    for attempt in range(max_retries):
        resp = httpx.get(url, headers=headers, timeout=30.0)

        if resp.status_code == 200:
            return resp.json()

        if resp.status_code in (429, 403):
            reset = resp.headers.get("X-RateLimit-Reset")
            wait = max(int(reset) - int(time.time()) + 1, 0) if reset else (2 ** attempt) + random.uniform(0, 1)
            time.sleep(min(wait, 60))
            continue

        if resp.status_code == 401:
            raise HTTPException(401, "Token invalid or revoked")

        raise HTTPException(resp.status_code, f"GitHub API error: {resp.text[:200]}")

    raise HTTPException(429, "Rate limit exhausted after max retries")
```

**Interview story for jitter:** "Exponential backoff without jitter causes thundering herd
(all retries fire at the same moment). Jitter adds `random.uniform(0,1)` to spread them.
The `X-RateLimit-Reset` header is better than exponential backoff — GitHub tells you
exactly when the window resets."

**Done when:** `/issues?repo=owner/repo` returns paginated issues. Hit the endpoint 100
times → it backs off gracefully instead of 429-flooding.

**Performance note — blocking sleep in workers:** `time.sleep(wait)` blocks the RQ worker
process for up to 60 seconds during rate-limit waits. One worker = one job at a time.
For the demo: run 2 workers (`rq worker zapbridge -w 2`) so webhook jobs aren't delayed
by an issues-fetch that's waiting on a rate limit.
In production: separate queues — `queue = Queue('zapbridge')` for webhooks,
`issues_queue = Queue('issues-fetch')` for API polling. Different workers, no contention.
**Interview story:** "RQ workers are synchronous processes. A 60-second sleep in one job
blocks that worker for all other jobs. In production I'd use separate queues with
dedicated worker pools — webhook processing and API polling should never share a worker."

def list_failed_jobs():
    from rq import Queue
    from app.connections import redis_conn  # reuse shared connection, don't create a new one
    failed = Queue("failed", connection=redis_conn)

check."

### Rate-Limit Retry Loop (Slice 7)

```
fetch_with_retry(url, headers, max_retries=5)
         │
         ▼
    attempt = 0
         │
    ┌────▼─────────────────────────────────────────────────────────────┐
    │  GET url                                                          │
    │       │                                                           │
    │       ├── 200 ──────────────────────────────────────────► return json
    │       │                                                           │
    │       ├── 429 / 403 ──► read X-RateLimit-Reset header            │
    │       │                      │                                   │
    │       │                 reset present? ──► wait = reset - now    │
    │       │                 no header?      ──► wait = 2^attempt + jitter
    │       │                      │                                   │
    │       │                 sleep(min(wait, 60))                     │
    │       │                 attempt += 1                             │
    │       │                 loop ◄─────────────────────────────────  │
    │       │                                                           │
    │       ├── 401 ──────────────────────────────────── raise HTTP 401│
    │       │                                                           │
    │       └── other ────────────────────────────────── raise HTTP N  │
    │                                                                   │
    │  attempt >= max_retries ──────────────────────── raise HTTP 429  │
    └───────────────────────────────────────────────────────────────────┘
```
